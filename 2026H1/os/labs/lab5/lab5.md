# Лабораторная работа №5

Форма сдачи лабораторной работы:
- отчет (текстовый документ в свободном оформлении) **с результатами по каждому шагу**;
- демонстрация работы системы.

## Инцидент

Ночью поступает инцидент:

> «Сервис начал работать медленно. Иногда зависает. Пользователи жалуются на долгие ответы API.»

Особенности ситуации:

* централизованный мониторинг (например, Prometheus) отсутствует;
* доступ к установке новых инструментов ограничен;
* есть только стандартные утилиты Linux;
* проблема возникает периодически и не всегда воспроизводится.

## Ваша задача

Вам необходимо:

Разработать локальный инструмент, позволяющий в реальном времени выявлять процессы, перегружающие систему, и фиксировать эти события.

## Функциональные требования

Система должна:

* периодически получать список процессов;
* выявлять процессы с высокой нагрузкой (CPU / память);
* классифицировать ситуацию (WARNING / CRITICAL);
* логировать события;
* поддерживать управление через сигналы.

## Нефункциональные требования

* использовать только bash и стандартные утилиты;
* решение должно состоять из нескольких процессов;
* использовать механизмы IPC:
    * pipe;
    * FIFO;
    * сигналы.
* не писать один большой скрипт.

## Архитектурное требование

Решение должно быть построено как pipeline:

```text
[watcher] → [analyzer] → [alerter] → [logger]
```

Каждый компонент:

* отдельный процесс;
* взаимодействие через IPC.

## Подготовка среды

Чтобы кейс был воспроизводим:

```bash
yes > /dev/null &
```

Подсказка:

* этот процесс будет создавать нагрузку;
* его необходимо обнаружить.

## План выполнения

### Шаг 1. Первичная диагностика

#### Задача

Изучить текущие процессы и понять структуру вывода.

##### Команда

```bash
ps aux
```

##### Подсказки

Обратите внимание на столбцы:

* `$2` → PID
* `$3` → %CPU
* `$4` → %MEM
* `$11` → команда

#### Результат

Необходимо понять:

* как выглядит список процессов;
* какие поля нужны для анализа.

### Шаг 2. Фильтрация процессов (анализ нагрузки)

#### Задача

Вывести процессы с высокой загрузкой CPU.

##### Подсказка

```bash
ps aux | awk '$3 > 20'
```

Оставить только нужные поля:

```bash
ps aux | awk '$3 > 20 {print $2, $3, $4, $11}'
```

#### Что происходит

* `ps` → один процесс;
* `awk` → второй процесс;
* pipe → IPC.

### Шаг 3. Реализация watcher

#### Задача

Сделать непрерывный мониторинг.

##### Шаблон

```bash
#!/bin/bash

while true; do
  ps aux
  sleep 2
done
```

#### Подсказка

Добавьте фильтрацию внутрь цикла.

### Шаг 4. Выделение analyzer (разделение ответственности)

#### Задача

Вынести логику анализа в отдельный процесс.

##### analyzer.sh (шаблон)

```bash
#!/bin/bash

while read line; do
  cpu=$(echo $line | awk '{print $3}')

  if (( $(echo "$cpu > 20" | bc -l) )); then
    echo "$line"
  fi
done
```

##### watcher.sh

```bash
#!/bin/bash

ps aux | ./analyzer.sh
```

#### Подсказка

Теперь у вас два процесса, которые взаимодействуют через pipe.

### Шаг 5. Переход к независимым процессам (FIFO)

#### Задача

Разделить процессы полностью.

##### Создание канала

```bash
mkfifo monitor.pipe
```

##### watcher пишет

```bash
ps aux > monitor.pipe
```

##### analyzer читает

```bash
cat monitor.pipe | ./analyzer.sh
```

#### Подсказка

Теперь процессы независимы, IPC реализован через ОС.

### Шаг 6. Добавление logger

#### Задача

Сохранять события в файл.

##### logger.sh (шаблон)

```bash
#!/bin/bash

while read line; do
  echo "$(date) | $line" >> monitor.log
done
```

##### Пример запуска

```bash
cat monitor.pipe | ./analyzer.sh | ./logger.sh
```

### Шаг 7. Добавление alerter

#### Задача

Классифицировать нагрузку.

##### Логика

* CPU > 20 → WARNING
* CPU > 50 → CRITICAL

##### Шаблон

```bash
cpu=$(echo $line | awk '{print $3}')
pid=$(echo $line | awk '{print $2}')
```

### Шаг 8. Управление через сигналы

#### Задача

Сделать корректное завершение.

##### watcher.sh (добавить)

```bash
running=true

trap "echo stopping; running=false" SIGTERM
trap "echo reset" SIGUSR1

while $running; do
  ps aux
  sleep 2
done
```

#### Проверка

```bash
kill -SIGTERM <pid>
```

### Шаг 9. Анализ системных вызовов

#### Задача

Посмотреть, как работает система внутри.

##### Команда

```bash
strace -p <pid>
```

##### Найти

* read
* write

#### Подсказка

Это и есть механизм IPC.

---

## Финальная сборка системы

```bash
mkfifo monitor.pipe

./logger.sh < monitor.pipe &
./watcher.sh > monitor.pipe
```

## Ожидаемый результат

Система должна:

* находить «тяжёлые» процессы;
* выводить предупреждения;
* логировать события;
* корректно завершаться.
