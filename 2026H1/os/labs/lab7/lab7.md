# Лабораторная работа №7

Форма сдачи лабораторной работы:

- отчет (текстовый документ в свободном оформлении) **с результатами по каждому этапу и пояснениями**.

---

## Сценарий

Вы — системный администратор небольшой компании. К вам пришла задача: подготовить сервер для новой команды, которая
будет разрабатывать веб-приложение. В команде есть два разработчика, дизайнер и автоматический деплой-бот. У каждого —
разные права на разные части проекта. Ваша задача — настроить всё правильно с нуля.

---

## Этап 1. Пользователи и группы

Создайте следующих пользователей:

| Пользователь | Тип       | Оболочка            |
|--------------|-----------|---------------------|
| `developer1` | обычный   | `/bin/bash`         |
| `developer2` | обычный   | `/bin/bash`         |
| `designer`   | обычный   | `/bin/bash`         |
| `deploy-bot` | системный | `/usr/sbin/nologin` |

Создайте группы: `dev-team`, `design-team`, `deployers`.

Добавьте пользователей в группы:

- `developer1`, `developer2` → `dev-team`
- `designer` → `design-team`
- `deploy-bot` → `deployers`

**Проверьте результат** командой `id` для каждого пользователя и убедитесь, что группы назначены верно.

---

## Этап 2. Структура каталогов

Создайте каталог `/srv/webapp/` со следующей структурой:

```
/srv/webapp/
├── src/        — исходный код
├── public/     — статические файлы (HTML, CSS, картинки)
├── config/     — конфигурационные файлы
├── logs/       — логи приложения
└── deploy/     — скрипты деплоя
```

Для каждого каталога назначьте владельца и группу согласно таблице:

| Каталог   | Владелец     | Группа      |
|-----------|--------------|-------------|
| `src/`    | `developer1` | `dev-team`  |
| `public/` | `developer1` | `dev-team`  |
| `config/` | `root`       | `dev-team`  |
| `logs/`   | `root`       | `dev-team`  |
| `deploy/` | `deploy-bot` | `deployers` |

**Проверьте результат** командой `ls -la /srv/webapp/`.

---

## Этап 3. Права доступа

Настройте права на каталоги согласно требованиям:

**`src/`** — разработчики читают и пишут, дизайнер только читает, остальные не имеют доступа:

```
dev-team:   rwx
design-team: r-x  ← нужно решить через ACL
other:      ---
```

**`public/`** — разработчики и дизайнер читают и пишут, остальные только читают:

```
владелец:  rwx
группа:    rwx
other:     r-x
```

**`config/`** — только `root` пишет, разработчики читают, остальные не имеют доступа:

```
root:      rwx
dev-team:  r-x
other:     ---
```

**`logs/`** — все пользователи могут создавать файлы, но удалять — только свои:

```
владелец:  rwx
группа:    rwx
other:     rwx
+ sticky bit
```

**`deploy/`** — `deploy-bot` читает и пишет, разработчики только читают, остальные не имеют доступа:

```
владелец:  rwx
группа:    r-x  ← dev-team нужно добавить через ACL
other:     ---
```

После настройки проверьте каждый каталог через `ls -la` и `getfacl`.

## Этап 4. Специальные биты

### SGID на `src/`

Установите SGID на каталог `src/`.

Затем войдите под `developer1` и создайте файл:

```bash
su - developer1
touch /srv/webapp/src/test.py
ls -l /srv/webapp/src/test.py
```

Посмотрите на группу созданного файла.

### Sticky bit на `logs/`

Установите sticky bit на каталог `logs/`.

Проверьте поведение:

1. Войдите под `developer1` и создайте файл `dev1.log`.
2. Войдите под `developer2` и попробуйте удалить `dev1.log`.
3. Зафиксируйте результат.

## Этап 5. Диагностика и sudo

### Диагностика

У вас намеренно сломано три сценария. В каждом случае при попытке выполнить операцию вы получите
`Permission denied`. Найдите причину и исправьте её.

**Сценарий А.** Пользователь `designer` не может прочитать файл `/srv/webapp/src/styles.css`, хотя права на сам файл
выставлены как `r--r--r--`.

**Сценарий Б.** Пользователь `developer2` только что был добавлен в группу `deployers`, но всё ещё не может прочитать
содержимое `/srv/webapp/deploy/`. Команда `id` показывает, что он в группе. Почему доступ всё равно закрыт и как это
исправить без перезагрузки системы?

**Сценарий В.** `deploy-bot` не может записать файл в `/srv/webapp/deploy/`, хотя является владельцем каталога. Найдите
причину через `stat` и `ls -la`.

Для диагностики используйте последовательность:

```
id → ls -la → namei -l → getfacl → stat
```

Зафиксируйте в отчёте каждый шаг: команда — вывод — вывод о причине.
