# Межпроцессное взаимодействие (IPC)

## Проблема: изоляция процессов

Операционная система обеспечивает изоляцию процессов, что означает:

* отсутствие доступа к памяти других процессов
* невозможность прямого управления другими процессами

Данная изоляция необходима для обеспечения:

* безопасности
* стабильности системы

В то же время современные программные системы требуют взаимодействия между процессами:

* браузер ↔ вкладки
* клиент (frontend) ↔ сервер (backend)
* сервис ↔ сервис

Возникает фундаментальный вопрос:

> Каким образом изолированные процессы могут взаимодействовать?

Для решения данной задачи используются механизмы межпроцессного взаимодействия – **IPC (Inter-Process Communication)**, которые обеспечивают:

* обмен данными
* координацию выполнения процессов

## Современные приложения как совокупность процессов

Современные программные системы, как правило, состоят из множества процессов.

### Пример: веб-браузер

Браузер включает несколько специализированных процессов:

* пользовательский интерфейс (UI)
* процессы вкладок
* сетевой процесс
* графический процесс (GPU)

### Причины такого подхода:

* обеспечение изоляции компонентов
* повышение отказоустойчивости системы

### Роль IPC:

IPC используется для:

* передачи данных между процессами
* обработки событий
* передачи управляющих команд

## IPC в backend-разработке

Типичная архитектура распределённого приложения может быть представлена следующим образом:

```
Frontend → Backend → Database
```

Такая система характеризуется тем, что:

* компоненты реализованы в виде отдельных процессов
* процессы могут быть распределены по различным узлам сети

### Используемые механизмы IPC:

* HTTP (на основе сокетов)
* gRPC
* очереди сообщений

## IPC в задачах DevOps

В операционных системах семейства UNIX широко используются конвейеры обработки данных.

Пример:

```bash
cat logs.txt | grep ERROR | wc -l
```

В данном случае:

* каждая команда выполняется в отдельном процессе
* процессы взаимодействуют посредством каналов (pipes)

## IPC в контейнеризации и микросервисной архитектуре

Контейнеризация обеспечивает изоляцию процессов, однако внутри контейнера процессы продолжают взаимодействовать.

### Используемые механизмы IPC:

* сокеты
* разделяемая память (shared memory)

## IPC в графических интерфейсах

Даже базовые пользовательские действия, такие как клик мыши, реализуются через цепочку взаимодействий:

* устройство ввода → драйвер
* драйвер → ядро операционной системы
* ядро → пользовательский процесс

Таким образом, обработка событий ввода также основана на механизмах IPC.

## IPC в высоконагруженных системах

В системах с высокими требованиями к производительности применяются специализированные механизмы межпроцессного взаимодействия:

* разделяемая память (shared memory)
* технологии zero-copy

## Классификация IPC

Механизмы межпроцессного взаимодействия можно классифицировать по нескольким основаниям, в зависимости от способа передачи данных, характера взаимодействия и области применения.

### По способу передачи данных

С точки зрения организации обмена данными выделяются два принципиально разных подхода:

* потоковая передача данных
* разделяемая память

При потоковой передаче один процесс формирует поток данных, который передаётся другому процессу. Такой подход используется в каналах и сокетах.

Пример:

```bash
cat logs.txt | grep ERROR
```

В данном случае:

* `cat` читает файл и передаёт данные
* `grep` принимает поток и фильтрует его

Данные проходят через pipe и копируются между процессами.

В случае разделяемой памяти процессы работают с одной областью памяти:

* данные не передаются – они уже находятся в общей области
* оба процесса читают и записывают одни и те же данные

Пример сценария:

* один процесс записывает данные (например, обновляет структуру)
* другой процесс читает эти данные без копирования

Это используется, например, в высоконагруженных системах обработки данных.

### По синхронности взаимодействия

IPC может быть **синхронным** или **асинхронным**.

Синхронное взаимодействие означает, что процесс ожидает результат операции.

Пример:

```bash
read input
```

Процесс блокируется, пока пользователь не введёт данные.

Асинхронное взаимодействие позволяет процессу продолжать выполнение.

Пример:

```bash
sleep 1000 &
```

Процесс запускается в фоне:

* основной процесс продолжает работу
* взаимодействие происходит через сигналы или другие механизмы

### По области применения

IPC может быть **локальным** или **сетевым**.

Локальный IPC работает в пределах одной системы.

Пример:

```bash
ls | wc -l
```

Сетевой IPC используется между разными машинами.

Пример:

```bash
curl http://example.com
```

Здесь:

* клиентский процесс обращается к серверному
* взаимодействие происходит через сокеты

## Основные механизмы IPC

### Сигналы

Сигналы используются для передачи событий между процессами и ядром операционной системы. Они представляют собой асинхронные уведомления, которые могут изменить поведение процесса.

Их особенности:

* не передают данные (только факт события)
* доставляются асинхронно
* используются для управления и уведомления
* могут быть обработаны процессом (handler) или привести к действию по умолчанию

#### SIGTERM – корректное завершение

Сигнал по умолчанию для завершения процесса.

```bash
kill -SIGTERM <pid>
```

или сокращённо:

```bash
kill <pid>
```

Поведение:

* процесс получает сигнал
* может корректно завершиться (закрыть файлы, сохранить состояние)

#### SIGKILL – принудительное завершение

Невозможно перехватить или проигнорировать.

```bash
kill -SIGKILL <pid>
```

Поведение:

* процесс немедленно уничтожается
* очистка ресурсов не выполняется

Используется:

* если процесс «завис»

#### SIGINT – прерывание с клавиатуры

Генерируется при нажатии Ctrl+C.

Пример:

```bash
sleep 1000
```

Нажать:

```
Ctrl + C
```

Поведение:

* процесс прерывается
* обычно завершает работу

#### SIGSTOP – остановка процесса

Полностью приостанавливает выполнение.

```bash
kill -SIGSTOP <pid>
```

Поведение:

* процесс «замораживается»
* не выполняется, но не завершён

#### SIGCONT – продолжение выполнения

Возобновляет остановленный процесс.

```bash
kill -SIGCONT <pid>
```

Поведение:

* процесс продолжает выполнение

#### SIGTSTP – остановка с клавиатуры

Аналог SIGSTOP, но может быть обработан.

Пример:

```bash
sleep 1000
```

Нажать:

```
Ctrl + Z
```

Поведение:

* процесс переводится в background и останавливается

Проверка:

```bash
jobs
```

#### SIGHUP – «перезапуск» или потеря терминала

Изначально означал разрыв соединения с терминалом.

```bash
kill -SIGHUP <pid>
```

Современное использование:

* перезагрузка конфигурации сервиса

Пример:

* nginx перечитывает конфигурацию

#### SIGUSR1 / SIGUSR2 – пользовательские сигналы

Предназначены для произвольного использования программами.

```bash
kill -SIGUSR1 <pid>
```

Поведение:

* определяется самим приложением

Примеры:

* включение логирования
* переключение режима работы

### Каналы (pipes)

Каналы обеспечивают передачу данных в виде потока.

Основные свойства:

* однонаправленность
* последовательная передача
* FIFO

Пример:

```bash
ps aux | grep python
```

Один процесс передаёт список процессов, другой фильтрует.

### Именованные каналы (FIFO)

FIFO позволяют взаимодействовать независимым процессам.

Пример:

```bash
mkfifo mypipe
```

Терминал 1:

```bash
cat mypipe
```

Терминал 2:

```bash
echo "hello" > mypipe
```

Здесь:

* процессы не связаны
* но обмениваются данными через файловую сущность

### Разделяемая память (shared memory)

Разделяемая память используется, когда важна производительность.

Пример сценария:

* процесс A записывает массив данных
* процесс B анализирует его

При этом:

* данные не копируются
* используется одна область памяти

Типичный пример применения:

* игровые движки
* системы реального времени

### Сокеты (sockets)

Сокеты обеспечивают универсальный механизм взаимодействия.

Пример (локальный):

```bash
nc -l 1234
```

```bash
nc localhost 1234
```

Пример (сетевой):

```bash
curl http://example.com
```

Сокеты используются:

* в backend-разработке
* в клиент-серверных системах

## Выбор механизма IPC

Выбор механизма определяется задачей.

Типичные сценарии:

* обработка данных в цепочке

  ```bash
  cat file.txt | grep error | wc -l
  ```

  → используется pipe

* взаимодействие независимых процессов
  → FIFO или socket

* высоконагруженные системы
  → shared memory

* взаимодействие по сети

  ```bash
  curl http://api.service
  ```

  → socket

* управление процессом

  ```bash
  kill -SIGKILL <pid>
  ```

  → signals

Таким образом, выбор IPC зависит от требований к:

* производительности
* сложности
* области применения

## Связь IPC с другими компонентами ОС

IPC не существует изолированно и тесно связан с другими подсистемами.

Он опирается на модель процессов:

* процессы являются участниками взаимодействия

Реализуется через системные вызовы:

* например, `read`, `write`, `kill`

Работает под управлением ядра:

* ядро контролирует доступ
* обеспечивает безопасность

Зависит от планировщика:

* определяет, когда процессы выполняются
* влияет на задержки взаимодействия

Пример:

```bash
cat file.txt | grep error
```

Здесь:

* создаются два процесса
* ядро управляет их выполнением
* планировщик определяет порядок работы
* IPC связывает их через pipe
